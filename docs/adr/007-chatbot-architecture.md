# ADR-007: Chatbot Architecture with Full RAG and Guardrails

**Status**: Proposed

**Date**: 2026-06-04

## Context

The platform includes a conversational chatbot that answers questions from extracted structured data, searches source documents, and uses NER model inference. Requirements mandate: tenant-scoped responses, source citations with traceability, hallucination guardrails, and constraint to tenant data only. Target P95 response time <10 seconds.

We evaluated three architectures:

- **(a) Pure SQL query generation**: LLM generates SQL from natural language, executes against extracted entity tables. Simple but lacks document context and NER capabilities.
- **(b) Full RAG pipeline with structured data + document search + NER**: Combines SQL queries over extracted entities, semantic search over document chunks (pgvector), and real-time NER inference on relevant snippets.
- **(c) Fine-tuned LLM per tenant**: Maximum customization but prohibitive cost and operational complexity for a multi-tenant platform.

## Decision

**Use full RAG pipeline combining three data sources** (strategy b):

| Data Source | Technology | Purpose |
|---|---|---|
| Structured entities | PostgreSQL via controlled SQL generation | Aggregate queries over extraction results |
| Document search | pgvector (semantic + full-text) | Retrieve relevant document snippets |
| NER model inference | Model Serving Layer | Real-time entity resolution on document context |

Guardrails:
- SQL validation layer — prevents destructive or unauthorized queries; executes in read-only transactions with timeouts.
- System prompt constrains LLM to tenant-scoped data sources.
- Source citation enforcement — responses without citations are rejected.
- Blocked question types (classification, generation, summarization of non-extracted data) return graceful message.
- Disclaimer included in every response.
- Rate limiting per tenant.
- Query complexity limits — block multi-hop questions requiring >3 source lookups.

LLM: OpenAI GPT-4o-class with tenant-scoped API keys. Evaluate open-source self-hosting for production scale.

## Consequences

### Positive
- Rich answers combining structured data, document evidence, and live NER.
- Source citations build user trust and enable verification.
- Guardrails prevent hallucination and data leakage.
- pgvector avoids operational complexity of a separate vector database.

### Negative
- Three-source RAG pipeline increases query latency (target P95 < 10s).
- SQL generation from natural language can produce incorrect queries — validation layer essential.
- LLM API cost scales with query volume.
- OpenAI dependency introduces external latency and availability risk.

### Mitigations
- Query caching for common question patterns.
- SQL validation layer in read-only transactions with timeouts.
- Fallback response if any RAG source is unavailable (graceful degradation).
- Evaluate open-source LLM self-hosting for production scale.
- Query complexity limits.

## Compliance

- Every chatbot response MUST include `sources` array with at least one citation.
- SQL queries generated by chatbot MUST be validated by the SQL validation layer before execution.
- Chatbot MUST NOT answer questions outside its defined capability list.
- All chatbot interactions MUST be tenant-scoped — no cross-tenant data access.
- Response latency MUST be monitored (P95 alert at >10s).
- Disclaimer MUST be included in every response.
